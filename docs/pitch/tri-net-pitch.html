TRI-NET · граничный радио-меш + ИИ на чипе

Один примитив без умножителя строит радио и ИИ на одном чипе за $100.

Троичные веса превращают каждое умножение в выбор знака (+x / −x / 0), поэтому тяжёлая по весам математика стоит ноль DSP. Это освобождает всю DSP-колонку Zynq-7020 — и софт-радио вместе с BitNet-класс трансформером живут на одном кристалле, в полностью открытом флоу, без Vivado.

примитив

Троичный вес — это не умножение. Это выбор.

Код согласованного фильтра, PN-последовательность спред-спектра и вес BitNet — все по природе троичны. Умножить отсчёт на {−1, 0, +1} не нужен DSP48 — только смена знака или ноль. Коррелятор сворачивается в дерево сумматоров; нейрослой — тоже.

Чистые данные по эфиру

BER 0 / 576 бит

Строка «TRINET ternary OTA link» передана как DSSS-BPSK и восстановлена точно — 3 из 3 кадров, ноль ошибок — декодировано M²-оценкой несущей + согласованным фильтром по 819-чиповой преамбуле + Costas-трекингом. Деспред — это и есть троичный коррелятор 0 DSP.

Захват кадра, проверено

89.6× пик/шум

Отношение захвата преамбулы в живом эфире. Негативный контроль — передатчик выключен — роняет RMS захвата 302 → 13 и замок 89.6× → 4.2×, значит байт нёс радио-линк, а не петля или артефакт обработки.

МЕШ

Два узла, одна полоса. Платы .11 и .13 передают одновременно в одном канале 2.4 ГГц разными PN-кодами; плата .12 извлекает каждое сообщение при BER = 0, разделяя только по коду — это CDMA multiple-access, на котором стоит меш. Near-far эффект проявляется ровно как в учебнике (сильный узел выигрывает; равная мощность требует контроля мощности или длиннее кодов) — измерено, не предположено.

ОДИН КРИСТАЛЛ

Радио и ОБУЧЕННЫЙ ИИ на одном чипе, вживую. AD9361 платы .12 ловит эфир на PL, а троичные модели работают на ARM того же чипа — все веса {−1,0,+1}. Обученная модель распознавания сети (99.8%) называет, кто в эфире, и управляет живым CSMA-MAC (слушать → передавать только когда чисто). Троичный трансформер работает на том же ядре (12/12 на attention-задаче), а троичный код-LM генерирует валидный код (fn main(){let x=phi*phi+1;}) — архитектура 91M IGLA-Coder в миниатюре, на узле.

Сеть, которая платит тем, кто её строит — $TRI DePIN

Один узел за ~$100 и говорит, и думает. Чтобы из узлов выросла сеть, держателям платят токен $TRI за реальную работу — релей чужого трафика по эфиру (модель Helium Proof-of-Coverage). Работа доказывается на самом узле: подделко-стойкий аккумулятор пересланных датаграмм, из которого слой расчётов чеканит $TRI. Написан на нашем языке t27 (specs/tri_depin.t27), проверен насквозь через golden pipeline: 14 инвариантов (нельзя раздуть счёт, переставить, переиграть, украсть чужую работу). Прогнан на реальном железе сквозняком: узел .13 отдаёт данные, узел .12 тянет их по сети и считает чек, подписывает Ed25519 своим ключом; host (x86) и второй узел .13 проверяют только по публичному ключу — бит-в-бит, без общего секрета. Инфляция счётчика и чужой ключ — отклонены. Расчётный слой (tri_settle.t27) делит пул $TRI пропорционально байтам, не превышая пул. Всё на ARM узла.

И это ложится в стандарт 6G: МСЭ-R IMT-2030 (июнь 2023) — 6 сценариев, и мы прямо в трёх — Integrated Sensing & Communication (наш RF-сенсинг/RTI), AI & Communication (ИИ на узле), Ubiquitous Connectivity (mesh без базовых станций). МСЭ-T SG-13, где как раз идёт архитектура Сети-2030, — площадка сотрудничества. Токен и открытый суверенный флоу — слой поверх стандарта: то, что запускает физическое развёртывание снизу.

Что НЕ сделано — прямо

  • Прошивка PS7-ядра на кремний. PS-управляемая интеграция собрана насквозь: hard-блок PS7 ведёт троичный MAC через EMIO GPIO, весь открытый флоу (yosys → nextpnr → fasm2frames → xc7frames2bit) доводит до загружаемого битстрима xc7z020 4.05 МБ на 308 МГц. Осталась только прошивка на кремний — с реальными пинами платы и мини-программой на PS; не прошито, т.к. pin-agnostic proof снял бы живое радио впустую.
  • 91M-модель — это архитектура + бюджет, проверенная по блокам и обученной малой моделью; на узле уже работают обученные троичные трансформер и код-LM в миниатюре — но полного прогона обучения на 91M ещё не было.
  • Радио-линк измерен на стендовой паре (антенна-в-антенну, 2.4 ГГц). Дальность, многолучёвка и полный меш узлов — следующий железный шаг; RX-калибровка двух плат (.11/.13) даёт «хороший замок / плохие данные» и требует настройки.
  • $TRI — радиоканал измерен, гейт под потери готов, полного OTA-переноса ещё нет. Байты идут по сети между узлами, чек подписан Ed25519 и проверяется публичным ключом; сам радиолинк .13→.12 измерен (SNR 47.4 дБ, BER≈0). глубина подбирается под пачку) + полный живой DePIN-раунд по РЕАЛЬНО измеренному покрытию (на ARM: 3 узла на реальных SNR-замерах 34/18/6 дБ, агрегация эпох + взвешенная выплата — узел 34 дБ получает 713 $TRI, 18 дБ — 258, 6 дБ — 27; живой Helium-класс Proof-of-Coverage) + Merkle-обязательство раунда (один корень над чеками, узел забирает награду пруфом — как Helium on-chain; подделка награды отклоняется) + живой 3-узловой релей .11→.12→.13 (оба релей-узла дают бит-в-бит одинаковый чек) + постоянный леджер $TRI (баланс копится между раундами, вся история в одном state-root — как цепочка состояний блокчейна; Все три — host-бит-в-бит; страж линка проверен на железе; повтор OTA отложен честно (линк на стенде физически просел). RRC-модем с приёмным матч-фильтром (полный цикл: RRC на TX + согласованный фильтр на RX + отсчёты в центрах символов → BER=0 «deadbeef», sync 63/63, ноль-ISI) + DSSS-код-разделение (CDMA) (два отправителя в ОДНОЙ полосе одновременно с разными PN-кодами; приёмник деспредит каждого своим кодом, чужой код — шум: оба BER=0 даже при шуме — много скрытных отправителей в одной полосе, дополнение к FDD) + само-диагностирующий стенд (свип TX+RX частота×усиление по метрике `linkq` НАШЁЛ рабочую точку и восстановил «мёртвый» линк: не антенна, а слабый сигнал — на RX-усилении 71 при 2.400/2.460 ГГц cp 0.5–0.65 «LINK OK», где фикс. 60 давал 0.1; линк маргинальный/фейдинг, но диагноз и восстановление — автоматически). Замкнутая адаптация линка на железе (узел сам выбирает RX-усиление по метрике cp и логирует качество во времени: телеметрия cp 0.11→0.56 за 8 с — фейдинг ИЗМЕРЕН, не заявлен, и ловится, cp=0.56 «LINK OK» на t5) + selection combining (для фейдинга: из сотен циклических копий взять лучшую по корреляции — host BER=0, на плате выбирает сильнейшую из ~233) + RLNC-кадры поверх CDMA (каждый источник спредит свои GF(256)-кодированные кадры своим PN-кодом в ОДНОЙ полосе; приёмник деспредит по кодам, проверяет MAC и решает: код A один → 0/1, A+B → 1/1 «TRINET-WIDEBAND!» даже при шуме — дальность DSSS × доступ CDMA × мультипуть RLNC в одном). Событийный триггер по cp на железе (декодировать ТОЛЬКО при удачном фейде: порог 0.85 → 30 попыток «жду фейд», caught=0, НОЛЬ мусора — честен по построению, не выдумывает декод) + системный бюджет ёмкости меша (ДОКАЗАНО: 768 кбод, 684 кбит/с нетто/линк, 3 FDD-полосы @2МГц → 1.64 Мбит/с агрегат; ПРОЕКЦИЯ по измеренному ACLR: 18 RRC-полос @1МГц → 9.85 Мбит/с, 72 код+частотных слота — честный раздел «доказано OTA / проекция»). Физика фейдинга измерена на железе (медленный фейдинг: автокорр. cp lag1 0.58–0.96 → длинные удачные окна; run2 = 98% кадров декодируемы) + ЧИСТЫЙ БАЙТ ПО ВОЗДУХУ ПОЙМАН (selection combining на восстановленной точке: try1 → 130 копий, best_cp=1.000, BER=0/32, recv=«deadbeef» — многоволновая OTA-дыра закрыта для одиночного источника).
