## Goal
Bring `FrameKind::Hello` beacons under authentication and enforce per-neighbor monotonic sequence + a bounded timestamp window, so replayed or forged HELLOs can no longer resurrect dead links or skew ETX history.

## Why / science
Today `src/discovery.rs::Hello` serializes as `[src:4][seq:4][n:1][heard:n├Ś4]` in the clear (min len 9), and `FrameKind::Hello` (`src/wire.rs`) exists but HELLOs bypass `Session::seal`/`open` (`src/crypto.rs`) entirely. The beacon carries only a plaintext `u32 seq` with **no freshness or replay check**. Once a HELLO ingest path feeds routing, an attacker (or a stale-buffer reflection) could replay an old HELLO to:
- resurrect a dead link ŌĆö `EtxTable::record(neighbor, we_heard_them, they_heard_us)` (`src/routing.rs`) is driven by whether we "heard" a HELLO this interval, so a replay would fabricate a reverse-link sample;
- skew the ETX history window and thus corrupt `best_next_hop()` routing decisions.

The 64-frame sliding replay window already protects the DATA plane (`src/crypto.rs::ReplayWindow`, `MeshError::Replay`); this issue extends the same guarantee to the control plane. This mirrors **SAODV** (Zapata & Asokan, *Securing ad hoc routing protocols*, ACM WiSe 2002, doi:10.1145/570681.570682), which signs the non-mutable fields of routing control messages to stop replay/forgery of topology state, and libsodium's replay-window pattern for per-peer monotonic counters.

## Scope note (read first)
`src/daemon.rs` today has **no HELLO send/receive path**; it only does `seal_data`/`open_data`, and `EtxTable::record` is never called outside routing unit tests. The full HELLO ingest loop is M2 work (`lib.rs`: M2 = TUN/netdev IP-over-radio with a real ETX metric, tri-net#11). **This PR therefore ships the authenticated HELLO *data structures + verifier*, not a live daemon loop.** The daemon-integration acceptance below is written against a minimal in-test HELLO handler so it does not depend on M2 landing. Gated on B01.

## Tasks
- [ ] **wire.rs** ŌĆö extend the HELLO on-wire format to `[src:4][seq:4][ts:8][n:1][heard:n├Ś4]`, adding an 8-byte big-endian `ts` (millis since a fixed epoch). Bump `wire::VERSION` 1 ŌåÆ 2. NOTE: this const is shared by the DATA `Header`, so both frame kinds move to v2 in lockstep ŌĆö acceptable because no node has ever been flashed (greenfield, no live compat break). `Header::parse` already rejects non-matching versions.
- [ ] **discovery.rs** ŌĆö add `ts: u64` to `Hello`; update `to_bytes`/`parse` (min length 9 ŌåÆ 17) and fix the three existing unit tests (`hello_roundtrips`, `empty_heard_list_ok`, `truncated_is_rejected`). Add a `HelloAuth` verifier holding per-neighbor `last_seq: u32` and `last_ts: u64`, with `check(&mut self, h: &Hello, now_ms: u64, skew_ms: u64) -> Result<(), MeshError>` that rejects non-monotonic `seq`, rejects `|now - ts| > skew_ms`, and only advances state on success.
- [ ] **crypto.rs** ŌĆö add `MeshError::Stale` for the freshness-window failure (distinct from `Replay`/`Auth`). Add a broadcast/HELLO MAC helper. **Key source must be specified:** HELLOs are 1ŌåÆN before any pairwise handshake, so the pairwise `Session` key is unusable. Use a **network-wide pre-shared key** (new `HelloKey` type, constructed from a 32-byte secret) ŌĆö do NOT invent a per-node identity keypair (no such primitive exists and no HMAC crate is in `Cargo.toml`). Implement the MAC with the existing `chacha20poly1305` crate: `ChaCha20Poly1305::new(hello_key).encrypt(fixed_nonce, {msg: b"", aad: serialized_hello_bytes})` and ship the 16-byte tag, matching the "serialized bytes are the AEAD associated data" pattern already documented in `wire.rs`. (If a keyed-hash MAC is preferred instead, add the `hmac` crate explicitly in this PR and say so.)
- [ ] **routing.rs** ŌĆö no metric-math change; add a doc note that `record` must only be reached for authenticated, fresh HELLOs.
