## Goal
After `k` consecutive missed HELLO beacons from a neighbor (detected via `Hello.seq` gaps), force that link's ETX to `INFINITY` and trigger a next-hop recompute immediately, instead of waiting for the ETX averaging window to decay.

## Why / science
`discovery.rs` already puts a monotonic `seq: u32` in every `Hello`, but nothing consumes it: `daemon::Node` holds an `EtxTable` yet never ingests HELLOs or their seq numbers, and `routing::LinkStats::etx()` only returns `INFINITY` once a direction's sliding window (`window`, e.g. 16) has decayed to a 0.0 delivery ratio — i.e. after up to `window` consecutive misses evict all prior `true` samples in one direction. So today a link death takes up to `window * beacon_interval` to be reflected in routing — an unbounded (window-scaled), untestable self-heal time. This is exactly the "self-healing convergence threshold UNDEFINED" greenfield risk from report v2.2 §4.

BFD (Bidirectional Forwarding Detection, RFC 5880) and the equivalent IGP "hello-multiplier" fast-failover pattern (OSPF/IS-IS, Cisco/Arista) solve this by declaring a neighbor down after a small fixed count of missed hellos rather than waiting on a coarse hold timer. Applying the same idea here: counting `seq` gaps lets us bound detection to `k * beacon_interval` — a concrete, testable number the DEMO GATE (M5 self-healing, tri-net) can assert against.

Citation: BFD, RFC 5880 (Katz & Ward, June 2010); IGP hello-multiplier fast-failover guidance (OSPF/IS-IS, Cisco/Arista).

## Tasks
- [ ] `src/discovery.rs`: add a small per-neighbor seq tracker (e.g. `SeqTracker`) that, given a neighbor's latest `Hello.seq`, returns the number of beacons missed since the last seen seq (handling first-sighting, in-order, and reordered/duplicate seqs; ignore wrap for now or document `u32` wrap as out of scope). Keep it a pure struct — no I/O.
- [ ] `src/routing.rs`: add `EtxTable::mark_dead(&mut self, neighbor: NodeId)` that forces the neighbor's link to `INFINITY` immediately (e.g. clear both delivery-ratio windows / set a sticky "dead" flag consulted by `LinkStats::etx()`), and clears on the next successfully recorded HELLO. This is the surgical hook that lets detection bypass the averaging window without changing existing ETX math.
- [ ] `src/routing.rs`: keep the missed-beacon threshold `k` configurable — accept `k` where the daemon calls it, or expose it on `EtxTable`; default derived from `window` (e.g. `k = 3`), do not hardcode magic numbers inline.
- [ ] `src/daemon.rs`: wire it in. On each received `Hello` (via a new `Node::on_hello(&mut self, hello: &Hello)` path — HELLO ingestion is currently absent from `Node`), update the seq tracker; when missed >= `k`, call `self.etx.mark_dead(src)` and recompute the next hop via existing `EtxTable::best_next_hop()`. On a fresh in-order HELLO, feed `etx.record(..)` as before so the link recovers.
- [ ] `src/wire.rs`: no format change required — `FrameKind::Hello` and `Header` already carry `src`; confirm the daemon dispatches `FrameKind::Hello` frames into `on_hello` (add the match arm if the receive path lands here).
- [ ] `src/crypto.rs`: no changes expected — HELLO authentication is out of scope for this PR; leave the session/AEAD path untouched (note it explicitly if touched).
- [ ] Update the M5 note in `src/lib.rs` doc comment only if the convergence bound becomes concretely representable (optional, doc-only).
