## Goal
Only admit a neighbor into the ETX routing table after a crypto session with it is established, and authenticate every HELLO beacon with a session-derived MAC, so a peer cannot forge `src`, inflate its metric, and blackhole traffic.

## Why / science
Today HELLOs are plaintext (`src/discovery.rs`: `Hello { src, seq, heard }` serialized in `to_bytes`/`parse` with no MAC), and `EtxTable::record` in `src/routing.rs` unconditionally `or_insert_with`s **any** `NodeId` it is handed (lines 97-105). That is the textbook AODV/OLSR false-metric attack: an attacker forges `src`, lists itself in every `heard` list to drive its forward and reverse delivery ratios to 1.0, wins `EtxTable::best_next_hop()` (ETX -> 1.0), then drops all forwarded traffic (blackhole).

SAODV-style protection fixes this by authenticating routing control messages so a node cannot originate or mutate a route advertisement without a key. Foundational work: Zapata & Asokan, "Securing ad hoc routing protocols," ACM WiSe 2002; a concrete SAODV-vs-blackhole treatment is IEEE doc [5376147](https://ieeexplore.ieee.org/document/5376147) ("SAODV: A MANET Routing Protocol that can Withstand Black Hole Attack"); routing-attack taxonomy in the survey [arxiv.org/abs/1407.3987](https://arxiv.org/abs/1407.3987). We bind HELLO integrity to the **existing per-peer session** established by the M1 handshake: an unauthenticated HELLO (no session, or bad MAC) never touches `EtxTable`.

**Scope honesty:** the current handshake (`src/crypto.rs`) uses `EphemeralSecret` **only** — there is no static/long-term identity key yet, so this PR does **not** claim full Sybil resistance. It closes the false-metric/blackhole surface by requiring a completed session before ETX admission. True Sybil binding (one static identity key = one `NodeId`, mixed into the handshake) is a **follow-up issue**, because it changes the handshake itself and cannot land cleanly in this PR.

## Tasks
- [ ] `src/crypto.rs`: derive a second key for HELLO MACs at session-creation time. `Session` currently stores only `cipher` — it discards the shared secret — so add the MAC key inside `from_shared`, where the shared bytes still exist: HKDF-expand with a **new** `HKDF_INFO_HELLO = b"hello-mac"` (distinct from the existing `HKDF_INFO = b"aead-key"`, so keys never collide) into a `hello_key: [u8; 32]` stored on `Session`. Expose `Session::hello_mac(&self, bytes: &[u8]) -> [u8; 16]` and `Session::verify_hello_mac(&self, bytes, tag) -> bool` (constant-time compare, e.g. via `subtle` or an inline constant-time equality). Use an HMAC-SHA256 truncated to 16 bytes, or reuse the ChaCha20-Poly1305 AEAD over empty plaintext with `bytes` as AAD.
- [ ] `src/discovery.rs`: extend the wire format to `[src:4][seq:4][n:1][heard: n×4][mac:16]`. Update `Hello::to_bytes`/`Hello::parse` to carry the 16-byte tag; add `Hello::authenticated(src, seq, heard, session) -> Vec<u8>` and `Hello::parse_verified(bytes, session) -> Option<Hello>` that recomputes and constant-time-compares the MAC over the `[src..heard]` prefix before returning. Reject on length or MAC mismatch.
- [ ] `src/routing.rs`: gate admission. Add `EtxTable::record_authenticated(neighbor, we_heard_them, they_heard_us)` as the only public entry that may insert a **new** `NodeId`; change the existing `record` so an **unknown** neighbor is dropped (return without inserting) — existing entries may still update. `best_next_hop()` can then only ever return an admitted peer. Update the two existing routing tests that seed via `record` to use the authenticated path (no behavior regression to ETX math).
- [ ] `src/daemon.rs` (`Node`): on HELLO receipt, look up the peer session in the existing `sessions` map; if `!self.has_session(src)`, drop the HELLO (no ETX update; optionally trigger a handshake). Only call `etx.record_authenticated(...)` after `Hello::parse_verified` succeeds against `self.sessions.get(&src)`.
- [ ] `src/wire.rs`: no format change to `Data`. Add a doc note on the `FrameKind::Hello` path that HELLO integrity is enforced in `discovery.rs` (session MAC), not the `Data` AEAD `open` path.
- [ ] Add a rate limit in `daemon.rs`: cap accepted HELLOs per authenticated `src` per interval (`MAX_HELLOS_PER_INTERVAL`), dropping excess to protect the endurance-limited flying node.
