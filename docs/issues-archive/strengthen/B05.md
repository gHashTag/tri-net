## Goal
Make ETX responsive to fast UAV topology change by replacing the uniform sliding-window delivery ratio in `src/routing.rs` with a WMEWMA estimator and an optimistic prior for fresh neighbors.

## Why / science
`DeliveryRatio::ratio()` in `src/routing.rs` is a uniform sliding-window mean (`received / samples.len()`): every sample in the window is weighted equally, so the estimate reacts on the timescale of the *whole* window. At ~12 m/s UAV cruise the topology changes faster than that averaging interval, so ETX is stale exactly when a link is degrading and a reroute is most needed. WMEWMA (weighted moving-average EWMA of packet reception rate) with `alpha ~ 0.3-0.5` keeps the estimate responsive to recent PRR while still filtering per-interval loss noise — the standard fix for mobile link-quality estimation.

Second defect: fresh neighbors. On the first few intervals `samples` is short or empty, `ratio()` returns `0.0`, and `LinkStats::etx()` collapses to `f32::INFINITY` — a just-discovered neighbor is treated as a dead link and excluded from `best_next_hop()` until the window fills. Bootstrap new links with an **optimistic prior** (assume good until proven bad) so they are usable immediately and demoted only on observed loss.

**Citations:** WMEWMA estimator — Woo, Tong & Culler, "Taming the underlying challenges of reliable multihop routing in sensor networks," SenSys 2003 (the primary source for WMEWMA). UAV-specific ETX staleness under mobility — Rosati et al., "Speed-Aware Routing for UAV Ad-Hoc Networks," arXiv:1307.6350. Broader ETX-family link-metric context — Javaid et al., arXiv:1311.3746 (general ETX metrics, not WMEWMA-specific).

## Tasks
- [ ] In `src/routing.rs`, replace the internals of `DeliveryRatio` with a WMEWMA estimator: keep the `record(got_hello: bool)` / `ratio() -> f32` API unchanged (drop-in), but compute the ratio as `est = alpha * sample + (1 - alpha) * est` per interval instead of a windowed mean.
- [ ] Add an `alpha: f32` field (default `0.4`, in `[0.3, 0.5]`) and an optimistic initial estimate (e.g. `est = 0.9`) set in `DeliveryRatio::new`, so a fresh neighbor's `etx()` is finite (~1.2), not `INFINITY`.
- [ ] Keep the existing `window: usize` constructor argument accepted by `DeliveryRatio::new` / `LinkStats::new` / `EtxTable::new` (do not break `EtxTable::new(etx_window)` — the only caller is `src/daemon.rs:32`); either derive `alpha` from `window` or ignore it with a doc note — no signature change to `EtxTable::new`.
- [ ] Preserve the dead-link semantics in `LinkStats::etx()`: a genuinely dead direction (sustained `record(false)`) must still decay the WMEWMA estimate toward `0.0` and cross a small epsilon so `etx()` returns `f32::INFINITY`. Introduce an explicit `DEAD_EPS` threshold instead of the `df <= 0.0` check. **Caution:** from the optimistic prior `0.9` at `alpha=0.4`, five `record(false)` intervals only decay `est` to `0.9*0.6^5 ≈ 0.07`; pick `DEAD_EPS` above that (or extend the dead-link test) so `dead_direction_is_infinite` still passes.
- [ ] Update the module doc-comment header in `src/routing.rs` (currently describes the flat forward/reverse delivery ratio) to describe WMEWMA + optimistic bootstrap. Do not touch `src/crypto.rs`, `src/discovery.rs`, or `src/wire.rs`.
- [ ] Update the existing tests in `src/routing.rs`: `perfect_link_is_etx_one`, `half_forward_doubles_etx`, `dead_direction_is_infinite`, `best_next_hop_picks_lowest_finite_etx` must still pass (feed enough intervals for the EWMA to converge within tolerance; widen tolerances where the EWMA steady state differs from the exact windowed mean).
