## Goal
Add a periodic HELLO beacon tick and a neighbor-expiry sweep to `src/daemon.rs` so dead/silent neighbors accumulate misses and decay, instead of only ever being updated when a HELLO is received.

## Why / science
`EtxTable::record()` in `src/routing.rs` is the **only** path that feeds the ETX sliding windows, but today it has **no production ingress caller at all** ‚Äî `src/daemon.rs` never calls it, so ETX is currently exercised only by `routing.rs` unit tests. The `Node` in `src/daemon.rs` has no time source, no beacon scheduler, and no expiry logic. Consequences once a HELLO ingress path is wired:
- A neighbor that goes silent (crashed, flew out of range, jammed) **never accumulates a miss** ‚Äî its `DeliveryRatio` window stops decaying and its ETX freezes at the last good value instead of climbing to `INFINITY`. `best_next_hop()` will keep handing traffic to a dead link.
- We never emit our own HELLOs on a schedule, so peers cannot compute their *forward* delivery ratio (`d_f`) to us ‚Äî the whole point of the `Hello.heard` list in `src/discovery.rs`.

The fix is a periodic **beacon tick** (send our HELLO) plus an **expiry sweep** that, each interval, calls `record(neighbor, we_heard_them=false, ‚Ä¶)` for every tracked neighbor from whom no HELLO arrived since the last tick. This makes silence decay the window exactly like a lost beacon ‚Äî the liveness-via-periodic-hello + detection-timeout pattern of **BFD (RFC 5880)**, and the HELLO-driven link-quality maintenance validated in FANET field tests by **Guillen-Perez et al. 2021 (doi:10.3390/APP11104363)**, where proactive HELLO upkeep is what lets ETX/Babel-class metrics react to topology change.

This is a **prerequisite** for the EWMA smoothing (B05) and seq-gap loss detection (B03) items: both need a regular, monotonic tick to reason about "misses per interval" ‚Äî B03 in particular needs the per-neighbor last-seen `seq` (already carried in `Hello.seq`, `src/discovery.rs`) to be observed every interval.

## Tasks
- [ ] In `src/daemon.rs`, add a `beacon_interval: Duration` and a per-interval `heard_this_interval: HashSet<NodeId>` (and/or a `last_seen: HashMap<NodeId, Instant>`) to `Node`. Use `std::time::{Duration, Instant}`; keep `#![forbid(unsafe_code)]` intact.
- [ ] Add `Node::on_hello(&mut self, hello: &discovery::Hello, now: Instant)` that: records the neighbor as heard this interval, updates `last_seen`, and calls `self.etx.record(hello.src, true, hello.reports_hearing(self.id))` ‚Äî making this the single HELLO ingress point (there is none today).
- [ ] Add `Node::tick(&mut self, now: Instant) -> Vec<u8>` that (a) builds our outgoing beacon via `discovery::Hello::new(self.id, seq, heard_list)` where `heard_list` is the neighbors heard since the previous tick and `seq` is a monotonic per-node counter, returns `hello.to_bytes()` for the `Transport` to send; and (b) runs the **expiry sweep**: for every neighbor tracked in `self.etx` **not** in `heard_this_interval`, call `self.etx.record(neighbor, false, false)` so the miss decays both windows. Clear `heard_this_interval` at the end of the tick.
- [ ] Inject the tick from a caller-driven clock (pass `now: Instant` in, do **not** spawn threads/`tokio`) so the scheduler stays deterministic and host-testable under `-sim`. Wire it so `smoke_m1.rs`-style host drivers can call `tick()` in a loop.
- [ ] Do not modify `src/crypto.rs` or `src/wire.rs`. `src/routing.rs` and `src/discovery.rs` should need no signature changes ‚Äî only new call sites from `daemon.rs`. Update the `Node` doc comment and the module docs in `src/lib.rs` (M2 line) only if a symbol name changes.

- [ ] **Measurable metric:** with `window = N`, a neighbor that stops beaconing reaches `ETX == INFINITY` within exactly `N` ticks and is excluded from `best_next_hop()` ‚Äî assert the miss/interval decay in `silent_neighbor_etx_decays_to_infinity`.
