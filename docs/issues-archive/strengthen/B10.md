## Goal
Add a periodic HKDF-chained ratchet (rekey every N frames, hard-capped below ChaCha20's per-key limit) and wrap all key material in `zeroize` so it is wiped on rekey and drop. Purely additive and host-testable (`-sim`); confined to `src/crypto.rs` + `src/wire.rs` + `Cargo.toml`. Daemon-loop wiring is explicitly deferred to M2 (see note).

## Why / science
`Session` in `src/crypto.rs` derives one AEAD key in `Session::from_shared` (HKDF-SHA256 over the X25519 shared secret, `HKDF_SALT`/`HKDF_INFO`) and uses it for the **entire session lifetime**, driving a monotonic `tx_counter: u64` through `seal`. Two problems:

1. **Per-key limit.** ChaCha20-Poly1305 is safe for up to ~2^32 blocks (~256 GiB) under one key/nonce prefix. A long-lived, high-throughput relay hop (backhaul carrying aggregated access traffic) can approach that ceiling; nonce/AEAD guarantees degrade past it.
2. **Long-lived key exposure.** A flying Mini node can be physically captured. With one key per session, capture leaks a key that decrypts the whole session (no forward secrecy within a session). The current code never explicitly zeroizes the HKDF output, the `EphemeralSecret` (`Handshake.secret`), or the derived key.

The fix is a symmetric ratchet: periodically derive `key_{i+1} = HKDF(chain_key_i)` (chain-key advance, ├Ā la Noise `Rekey()`), reset the counter/replay window, and zeroize the old key. Bounds data-per-key and gives forward secrecy across ratchet epochs at trivial cost (one HKDF-Expand per rekey).

**Citations:**
- Noise Protocol Framework spec, `Rekey()` and rekey guidance ŌĆö https://noiseprotocol.org/noise.pdf
- libsodium docs on the 2^32-block ChaCha20 per-key limit and rekeying.
- RustCrypto `zeroize` crate for wiping secrets on drop.

## Deferred to M2 (do NOT attempt in this PR)
`src/daemon.rs` is currently a bare `Node` skeleton over the `Transport` trait ŌĆö it has **no run loop and no periodic tick** (per its module doc, TUN/radio I/O and the event loop are M2 / tri-net#11). Therefore a **time-based** `T`-second ratchet trigger and daemon-side `MeshError::RekeyRequired` handling are out of scope here. Land only the frame-count-driven auto-ratchet + hard cap in `crypto.rs`; expose `pub fn ratchet(&mut self)` so the M2 loop can call it on a timer later.
