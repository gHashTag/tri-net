## Goal
Replace the current unauthenticated ephemeral-ephemeral handshake with a mutually-authenticated Noise-XX handshake over our existing X25519 + ChaCha20-Poly1305 + SHA-256 primitives, and cryptographically bind each peer to a long-term static key so a peer cannot be impersonated (MITM/Sybil). Authentication is anchored on the **full 32-byte static public key** (via an allow-list); `NodeId` is treated as a non-security-critical routing label derived from that key.

## Why / science
Today `Handshake` in `src/crypto.rs` uses `EphemeralSecret` on both sides (lines 29-40) and `complete()` does a raw `diffie_hellman` against **any** peer public key it is handed (lines 46-49). This is Noise-**NN**: unauthenticated ephemeral-ephemeral DH. `Session::from_shared` (lines 67-79) derives the AEAD key from that shared secret with no identity check, so an active attacker who substitutes their own ephemeral key completes a perfectly valid session — MITM and Sybil are indistinguishable from a real peer.

Every downstream defense we already built protects a channel to an *unauthenticated stranger*:
- The AEAD tag (`crypto.rs` `seal`/`open`, lines 83-130) proves integrity to *whoever we did DH with*, not to a known node.
- The replay window (`ReplayWindow`) stops replays *within* that unauthenticated channel.
- `daemon.rs::open_data` looks up a session by `src` `NodeId` (line 62), but `NodeId` is an unauthenticated `u32` from the cleartext `wire.rs` header (line 24) — nothing binds `src` to the key that terminated the handshake. Any node can claim any `NodeId`.
- ETX routing (`routing.rs`) then trusts HELLO beacons (`discovery.rs`, `Hello.src`) attributed to those spoofable ids.

This is the single largest crypto gap and the foundation for B02/B03/B04. libp2p runs Noise-XX over exactly our primitive set (X25519, ChaCha20-Poly1305, SHA-256), so the construction is well-specified and proven.

- Noise Protocol Framework, Perrin — https://noiseprotocol.org/noise.html (§7.5 XX pattern `-> e / <- e, ee, s, es / -> s, se`; §5.3 `HandshakeState`; §8 protocol naming; §12 crypto functions)
- libp2p Noise spec — https://github.com/libp2p/specs/blob/master/noise/README.md (`Noise_XX_25519_ChaChaPoly_SHA256`; MUST support 25519 + ChaChaPoly + SHA256; static-key → peer-id binding)

## Design note (security)
`NodeId` is a `u32` (`routing.rs` line 12). Deriving `NodeId = first 4 bytes of SHA-256(static_pub)` gives only ~2^16 collision resistance (birthday bound), so an attacker can grind keys to hit a target's `NodeId`. Therefore:
- The **authoritative** authentication decision uses the full 32-byte static `PublicKey` matched against an allow-list (`known_static_keys`), **not** the truncated `NodeId`.
- `NodeId` remains a convenient routing label; the `NodeId ↔ static-key` check is a fast pre-filter, never the sole gate. Document this truncation limitation in code.

- [ ] `crypto::tests::xx_handshake_mutual_auth_roundtrip` — both sides complete XX, derive the same AEAD key, exchange sealed frames, and each learns the other's static `PublicKey`.
- [ ] `crypto::tests::xx_rejects_static_key_not_in_allowlist` — a responder whose static key is not in `known_static_keys` yields `MeshError::AuthPeer`.
- [ ] `crypto::tests::mitm_substituted_static_is_rejected` — an active attacker that substitutes its own static key mid-handshake fails to complete against a peer with an allow-list (positive property of the new API; do NOT frame as a regression against the old NN code, which has no static keys and cannot express this test).

- [ ] `daemon::tests::open_rejects_spoofed_src_nodeid` — a frame whose header `src` differs from the session's bound key returns an error.
- [ ] Existing 20 unit + 2 integration tests + smoke stay green (`cargo test` + smoke binary).
