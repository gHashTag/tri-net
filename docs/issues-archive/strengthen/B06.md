## Goal
Add a multi-hop, loop-free routing table to `src/routing.rs`: additive path ETX plus a Babel-style per-source sequence + feasibility condition, so a node can pick the best next hop toward *any* destination — not just a direct neighbor.

## Why / science
Today `EtxTable::best_next_hop()` in `src/routing.rs` ranks only **direct** neighbors by single-link ETX. It has no concept of a route *toward a distant destination* and no loop-avoidance — yet a 3–8 node relay mesh exists precisely to forward traffic over multiple hops. Without a path metric, a 3-node triangle (M4) and the self-healing re-route (M5, tri-net DEMO GATE) cannot be expressed at all: there is no data structure that says "to reach node D, send to next-hop B at path-cost X."

Real FANET flight tests (Guillén-Pérez et al., *Applied Sciences* 2021, doi:10.3390/APP11104363) rank **Babel > OLSR > BATMAN-adv** for UAV mesh. Babel (RFC 8966) achieves cheap loop-freedom with a distance-vector **feasibility condition**: a received route update is only *feasible* — and only installed — if it is strictly fresher (higher per-source seqno) or equal-seqno with strictly lower advertised metric than the feasibility distance already recorded for that source. This gives loop avoidance without full link-state flooding, which matters on a bandwidth-starved half-duplex radio (each hop ~50% duplex loss per report v2.2). Path ETX is additive over hops, so a route's cost is `sum(link_etx)` and the best next hop minimizes total path ETX — consistent with the existing single-link ETX = 1/(d_f·d_r) already implemented and tested here.

Scope note: this is a **`-sim`** (host-tested) deliverable. It never touches hardware; the Mini node was not flashed as of 2026-07-01.

## Tasks
- [ ] In `src/routing.rs`, add a `RouteUpdate` value carrying `{ source: NodeId, seqno: u16, metric: u32 }` (advertised path ETX quantized to an integer; keep ETX `f32` internally, quantize only on the wire to keep updates comparable and cheap).
- [ ] In `src/routing.rs`, add a `RoutingTable` struct holding, per destination `source`: the selected `next_hop: NodeId`, the installed `metric` (path ETX from *us*), the `seqno`, and the **feasibility distance** (best metric ever advertised for that source at the current/known seqno).
- [ ] Implement `RoutingTable::feasible(&self, update, via_link_etx) -> bool`: accept iff `update.seqno` is strictly greater than the stored feasibility seqno, OR (`update.seqno` equal AND `update.metric + quantize(via_link_etx)` strictly less than the stored feasibility distance). Reject otherwise — this is the loop-free guarantee. Saturate on `u32` add so a dead/infinite link never wraps.
- [ ] Implement `RoutingTable::integrate(&mut self, from: NodeId, update: RouteUpdate, via_link_etx: f32)`: if feasible, compute `total = update.metric + quantize(via_link_etx)`, install `next_hop = from`, update feasibility distance/seqno, and return the change so the caller can decide whether to re-advertise.
- [ ] Implement `RoutingTable::next_hop(&self, dst: NodeId) -> Option<(NodeId, u32)>` returning the installed next hop and path metric for a destination.
- [ ] Wire the direct-link cost from the existing `EtxTable`: add `EtxTable::quantized_etx(neighbor) -> Option<u32>` (or a shared `quantize(f32) -> u32` helper) so `integrate` uses the *same* ETX already maintained from HELLO exchanges — do not introduce a second link metric.
- [ ] In `src/discovery.rs`, extend the HELLO beacon to piggyback route advertisements: append a route section `[m:1][ (source:4, seqno:2, metric:4) × m ]` after the existing `heard` list. Keep the existing `[src:4][seq:4][n:1][heard: n×4]` prefix byte-compatible; bump nothing in `wire.rs::VERSION` unless the `Header` layout itself changes (it does not — routes ride inside the HELLO payload, not the fixed header). Update `Hello::to_bytes`/`Hello::parse` and their round-trip/truncation tests accordingly.
- [ ] In `src/daemon.rs`, add a `RoutingTable` field to `Node` (alongside the existing `EtxTable`), a method to originate this node's own route (`source = self.id`, monotonically increasing `seqno`), and a method to process an incoming HELLO's route section into `RoutingTable::integrate`. Add `Node::route_to(dst) -> Option<(NodeId, u32)>` delegating to `RoutingTable::next_hop`.
- [ ] `seal_data`/`open_data` in `src/daemon.rs` stay unchanged in signature; add a doc line noting the caller now resolves `dst` → next hop via `route_to` before sealing (forwarding loop itself is M2/tri-net#11 and out of scope here). No change to `src/crypto.rs` or `src/wire.rs` header format.

## Acceptance
- [ ] `cargo test` green with new unit tests in `src/routing.rs`:
  - `path_etx_is_additive_over_two_hops` — A→B (etx 1.0) + B→C (etx 2.0) yields path metric to C == quantize(3.0) via next hop B.
  - `stale_seqno_update_is_infeasible` — an update with a lower seqno than stored is rejected; table unchanged.
  - `higher_metric_same_seqno_is_infeasible` — equal seqno, worse metric rejected (loop-free case).
  - `strictly_better_update_is_installed` — equal seqno, lower total metric installed and next_hop switches.
  - `fresher_seqno_always_feasible` — higher seqno accepted even if metric is worse (route repair).
  - `saturating_add_on_infinite_link` — integrating over an infinite/dead link does not panic or wrap; destination stays unreachable.
- [ ] `cargo test` green with updated `src/discovery.rs` tests: `hello_roundtrips_with_routes` and a truncated-route-section rejection test; existing `empty_heard_list_ok` still passes (zero routes).
- [ ] `cargo test` green with a new integration test (`tests/`): build a 3-node A–B–C line where A and C are **not** direct neighbors, run N HELLO rounds, and assert `A.route_to(C)` resolves to next hop B with metric == sum of the two link ETXs. This is the measurable metric.
- [ ] **Loop-freedom check (measurable):** a test that injects a route update which *would* form a 3-node cycle and asserts the feasibility condition rejects it, so no destination ends with a next hop that points back along its own installed path.
- [ ] `cargo clippy --all-targets -- -D warnings` clean; `#![forbid(unsafe_code)]` preserved.
- [ ] Sim-vs-hw marker: all tests run host-side (`-sim`); PR description states "never run on hardware — Mini unflashed as of 2026-07-01" and that real per-neighbor timing/loss graduates to `hw` at M2.

## Effort / Impact
Effort **M** (one focused PR, ~routing.rs + discovery.rs + daemon.rs, no crypto/wire-header changes). Impact **high** — unblocks the M4 triangle and M5 self-healing DEMO GATE; without a multi-hop table the relay mesh cannot route past one hop.
