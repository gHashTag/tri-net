# Merge runbook: landing the codegen fixes and the mesh specs

The autonomous work spans two repos. This is the exact, ordered sequence to land
it once authorized. Nothing here has been executed -- reseal and merge to master
need explicit approval. Read the caveats first.

## What is being landed

- **t27** (compiler, github/gHashTag/t27): branch `codegen-clean` is **112 commits
  ahead of `master`, 1 behind**. It carries 13 codegen/parser fixes (silent-drop
  audit + fixes #1-13) plus other numeric/verilog work. Because the fixes change
  generated output, 214 saved seals no longer match and must be re-sealed on the
  corrected output before merge.
- **tri-net** (product, github/gHashTag/tri-net): branch `feat/m2-hw-bringup`
  carries 17 mesh specs (13 FEC + 4 routing) + 4 architecture/audit docs. These
  specs are unsealed (tri-net keeps only 1 legacy seal and rebuilds `gen/` via
  build.rs), so tri-net needs only a normal branch merge -- no reseal.

## Part A -- t27: reseal then merge (the complex part)

1. **Confirm baseline on codegen-clean** (zero parse/gen failures; seal mismatches
   are the only fails):
   ```
   cd ~/Desktop/PROJECTS/CLAUDE/t27 && git switch codegen-clean
   $T27C suite --repo-root .     # expect: Parse/Gen failures 0, Seal mismatches ~214
   ```

2. **Re-seal every spec on the corrected output.** There is no bulk command; loop
   `seal --save` (safe: it only rewrites .trinity/seals/<module>.json):
   ```
   find specs -name '*.t27' -print0 | while IFS= read -r -d '' s; do
     $T27C seal "$s" --save
   done
   ```

3. **Remove stale duplicate seals.** The module-name scheme is mixed: 637 seals use
   a bare name (ternary_add.json) and 8 use a path-prefixed one
   (base_ternary_add.json). `seal --save` writes the prefixed name, so the bare
   duplicate goes stale. After resealing, delete any seal whose module no longer
   resolves, then re-run the suite -- it must show **0 seal mismatches**:
   ```
   $T27C suite --repo-root .     # expect: Seal mismatches 0, all phases pass
   ```

4. **Commit the reseal** (precedent: commit bf1f07e3 "name function-body blocks +
   reseal all specs"):
   ```
   git add .trinity/seals && git commit -m "chore(seals): reseal all specs on fixed codegen (fixes #1-13)"
   ```

5. **Reconcile the 1 master-only commit** before merging. It is a docs sync
   (`3c19bf08 docs(numeric): sync SSOT ...`), no code conflict expected:
   ```
   git switch master && git pull
   git switch codegen-clean && git merge master   # brings 3c19bf08 in, resolve if needed
   ```

6. **Merge to master.** This lands all 112 codegen-clean commits, not just the 13
   fixes -- confirm that scope is intended (see caveats). Open a PR per the repo's
   Issue Gate (do not self-approve):
   ```
   git switch master && git merge --no-ff codegen-clean
   ```

## Part B -- tri-net: plain branch merge

The 17 specs are already committed on `feat/m2-hw-bringup` and are unsealed, so:
```
cd ~/Desktop/PROJECTS/CLAUDE/tri-net
$T27C suite --repo-root . 2>/dev/null || true   # optional; gen is rebuilt by build.rs
git switch master && git merge --no-ff feat/m2-hw-bringup
```
A fresh `cargo build` (with the fixed t27c on PATH) regenerates `gen/` from the
specs; the property tests referenced in FEC_VERIFICATION.md re-run by wiring the
generated modules under `#[cfg(test)]`.

## Order and dependency

t27 first (Part A), then tri-net (Part B): tri-net's specs are generated by the
fixed t27c, so master t27c must carry the fixes before tri-net CI regenerates. If
tri-net pins a t27c version, bump it to the merged master commit.

## Caveats (read before running)

- **Reseal must precede merge**, never the reverse -- resealing stale/broken output
  would lock the pre-fix bugs into the seals. The 13 fixes make the output correct;
  reseal captures that.
- **Scope**: merging codegen-clean brings 112 commits. If only the 13 fixes + docs
  are wanted, cherry-pick them onto a fresh branch off master instead of the full
  merge, then reseal.
- **Seal-name duplicates** (base_ vs bare) can leave the suite reporting a mismatch
  for a module that has two seal files; delete the stale one and re-verify.
- **CI**: the compiler suite is 1411 passed / 5 baseline failures (jwt x3, ternary,
  trit_stdlib) -- these predate this work; do not treat them as regressions.
- **Approval**: opening PRs and merging to master are explicit human steps; this
  runbook stops at "open a PR".
